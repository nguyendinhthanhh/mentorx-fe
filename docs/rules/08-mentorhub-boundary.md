# 08 MentorHub Boundary

## Purpose
Define MentorHub as a workspace with clear boundaries.

## Core Rules
- MentorHub is a workspace only.
- MentorHub is not a duplicate of the public marketplace.
- MentorHub pages must focus on real mentor operations.
- Messages should keep MentorHub context instead of feeling detached or generic.

## Correct MentorHub Focus
- Overview
- My Proposals
- Active Contracts
- My Courses
- Schedule
- Earnings
- Reviews
- Messages
- Settings

## Discovery Boundary
- MentorHub must not become a second jobs marketplace by default.
- If recommended jobs exist, they must be backed by real matching logic and clearly presented as recommendations, not a full duplicate browse experience.

## Messages Rules
- Mentor messages should preserve MentorHub context.
- The messages experience should be inbox-first and usable on narrow screens.
- Do not force a desktop split view as the only workable interaction pattern.

## Dashboard Rules
- MentorHub overview actions must be useful and real.
- No fake insight cards.
- No fake charts.
- No marketplace duplication disguised as overview content.

## Navigation Rules
- MentorHub sidebar should remain stable across mentor routes.
- Route changes inside MentorHub should not drop the workspace frame unexpectedly.

## Do
- Treat MentorHub as a work console for approved mentors.
- Keep action paths close to proposals, contracts, earnings, and messages.
- Use direct mentor-business language.

## Do Not
- Fill MentorHub with public marketplace browsing clones.
- Add job discovery panels unless they are real and useful.
- Let MentorHub feel like a generic template dashboard.

## MVP Scope
- A focused mentor workspace with clear operational value.

## Future Scope / TODO
- If recommendations mature, define a strict recommended-jobs pattern instead of broad marketplace duplication.
